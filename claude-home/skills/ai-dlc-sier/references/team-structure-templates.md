# AIネイティブ組織構造テンプレート

ケンタウロス・パラダイムに基づく AI ネイティブ組織の設計パターン。規模別テンプレート、ARE 役割定義、移行ロードマップ。

## ケンタウロス・ポッド：原子単位

チェスに由来する「ケンタウロス」概念（人間+AI チーム > 単独人間 or 単独 AI）をソフトウェア組織に適用。
AI を「ツール」ではなく「協働パートナー」として構造の中心に据えたチーム。

### 中核コンポーネント（4 役割）

| 役割 | 機能 | 比喩 |
|---|---|---|
| **シニア・アーキテクト** | 戦略方向・システム境界・コンテキスト保持 | 脳（意図の設定） |
| **AI 信頼性エンジニア (ARE)** | AI 出力の完全性保証、ガードレール・Evals 構築 | 免疫システム |
| **プロダクト・ストラテジスト** | ユーザーニーズ・ビジネス成果、AI へのプロンプト | ナビゲーター |
| **エージェント・フリート** | コーダー/テスター/レビュワー/ドキュメンターの AI 群 | 筋肉（実行部隊） |

### 経済的命題

従来: ベロシティ向上 = 人員増加 = 調整コスト O(n^2) 増大（ブルックスの法則）
AI ネイティブ: 人数一定 + 計算リソース/コンテキスト増強 = 非線形スケーリング

4-6 人のポッドが AI エージェント群で従来 40 人規模部門相当のアウトプットを理論上実現。
ただし ARE による堅牢な品質保証が前提条件。

## AI 信頼性エンジニア (ARE) 詳細

### QA/SDET との比較

| 比較項目 | 従来 QA/SDET | ARE | ビジネス価値 |
|---|---|---|---|
| テスト対象 | 機能テスト、回帰テスト | Evals、幻覚チェック、モデル挙動検証 | レピュテーションリスク回避 |
| コード関与 | テストスクリプト、バグ報告 | 仕様・ガードレール所有、プロンプト調整 | コンプライアンス遵守保証 |
| インシデント対応 | バグ起票・差し戻し | 推論チェーンのデバッグ、コンテキスト修正 | MTTR 短縮 |
| スケーラビリティ | 線形（テスト増=人員増） | 指数関数的（自動評価は計算量でスケール） | 10x 品質を人員増なしで |
| 検証アプローチ | 要件との厳密一致 | 意味的類似性、ドリフト検知 | 長期的システム整合性維持 |
| マインドセット | 決定論的（白か黒か） | 確率論的（信頼区間、許容誤差） | 不確実環境下の迅速判断 |

### ARE の 2 つの戦略的価値

1. **「静かな失敗」の捕捉**: AI が書く「見た目は正しいがビジネスロジック上微妙に誤った」コードを、従来の単体テストでは検出困難。ARE が意味論的チェックを設計
2. **「人材の空洞化」への解**: AI がジュニア作業を担うと育成機会が消失。ARE はジュニアの新エントリーポイント（AI コードの批評・検証を通じてアーキテクチャを学習）

## 規模別テンプレート

### 小規模：シングル・ポッド（3-5 人）

- **適用**: スタートアップ、企業内の個別機能開発
- **構成**: Human Core（Architect 1 + ARE 1 + Strategist 1）+ Agent Fleet（Coder/Tester/Reviewer/Doc）
- **ワークフロー**: 円環的。Strategist → Architect → Agent Fleet（モブ・コンストラクション）→ ARE（デプロイ・ゲート）
- **運用**: ARE は全行手動テストではなく、テストカバレッジとリスクプロファイルをレビュー（Human-on-the-Loop）

### 中規模：クラスター/トライブ（3-4 ポッド）

- **適用**: 統一プロダクトラインに取り組む複数チーム
- **追加役割**: **AI ソリューション・アーキテクト**（ポッド間の架け橋、エージェント定義・API 統一を管理）
- **追加インフラ**: 共有エージェント・プラットフォーム（Knowledge Core + Tools + Central Eval Engine）
- **ポッド間連携**: 共有 Vector DB でコンテキスト瞬時共有、API プロトコル統一

### 大規模：エンタープライズ・エージェント・エコシステム

- **適用**: 全社規模の AI ネイティブ移行
- **追加組織**: **エージェント・ファクトリー**（プラットフォームチーム）— エージェントの作成・保守を工業化
- **ガバナンス**: AI Governance Council（倫理・コンプライアンス）
- **オーケストレーション層**: ID 管理、セキュリティ、レート制限、エージェント間プロトコル
- **構造**: 垂直方向の専門化（事業部ポッド）+ 水平方向の機能提供（全社共通セキュリティエージェント等）

## 従来型 vs AI ネイティブ 構造比較

| 特徴 | 従来型ピラミッド | ケンタウロス・ネットワーク |
|---|---|---|
| コミュニケーション | 階層的、高レイテンシ、伝言ゲーム | ネットワーク型、AI 介在の瞬時共有 |
| スケーリングコスト | O(n^2)（人増=会議増） | 線形/劣線形（エージェント増=計算コストのみ） |
| 意思決定 | 承認チェーンによる遅延 | Human サインオフ付き分散型決定 |
| 品質管理 | 最終工程での検査（QA） | 継続的検証（ARE + Evals）、シフトレフト極致 |
| 人間の役割 | 構文（Syntax）の生産者 | 意図（Intent）の管理者、論理の検証者 |

**コンウェイの法則の再定義**: エージェントが API と Vector DB で通信するため、「コミュニケーション構造」が流動的に。組織制約ではなく技術的必要性に基づくアーキテクチャ設計が可能に。

## チーム・オブ・チームズ実装

50 ポッドが 50 の互換性のないシステムを構築するのを防ぐ「結合組織」:

- **ツールの標準化**: ロギング/認証/デザインシステムをエージェントプラットフォームのシステムプロンプト+ツールレジストリで強制
- **ポッド間コンテキスト共有**: API 仕様変更を共有 Knowledge Core 経由で即時通知 → 統合地獄の防止
- **リーダーシップ転換**: 「指揮統制」→「注視するが手は出さない（Eyes on, Hands off）」。タスクではなくシステム全体をチューニング

## 移行ロードマップ（3 フェーズ）

### Phase 1: パイロット（1-3 ヶ月）

- 1 つのケンタウロス・ポッド（3-5 人）を結成
- モブ・エラボレーションの概念実証
- ARE を 1 名採用または育成
- 指標: MTTV（平均検証時間）、欠陥捕捉率

### Phase 2: プラットフォーム構築（4-9 ヶ月）

- エージェント・プラットフォーム（Vector DB、ツールレジストリ）構築
- 複数ポッドのコンテキスト共有を実現
- AI ソリューション・アーキテクトを任命
- 指標: ポッド間依存摩擦（統合バグ減少率）

### Phase 3: スケーリング（10 ヶ月〜）

- 部門全体をポッド体制に移行
- 完全なチーム・オブ・チームズ調整を実現
- ARE を標準エントリーレベル職に
- 指標: 従業員一人当たり収益の大幅向上
